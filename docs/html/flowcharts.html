Interactive Flowcharts

Open a diagram to see the compiler path, run path, each run-control function, or the artifact database structure in detail.

Core flows

Start with the two big lifecycle diagrams.

Compiler Flowchart

API entry

  1. compile_pipeline(pipeline.py)
  2. Read pipeline.py text
  3. Resolve configurations.yaml text
  4. Preview compile_pipeline_code
->

Load source

  1. Prepare a temporary Python module for the pipeline file
  2. Run the pipeline file so Python creates the decorated functions
  3. Each Queron decorator stores node settings on its function
  4. Walk through the loaded functions and find Queron nodes
  5. Convert each decorated function into an internal node record
->

Build spec

  1. Read the configuration text into a structured config object
  2. Choose the active target from the argument or config default
  3. Group the pipeline id, target, nodes, and notebook metadata
  4. Start validating and filling in missing runtime details
->

Validate graph

  1. Make sure the pipeline has a pipeline id
  2. Check duplicate node names
  3. Make sure artifact-producing nodes have unique output names
  4. Convert each output name into a safe DuckDB table name
  5. Find refs, sources, and lookups inside SQL templates
  6. Use refs and manual depends_on to build dependencies
  7. Replace template helpers with real table or relation names
  8. Check runtime variables are used only in safe SQL positions
  9. Validate node-specific settings like files, targets, and checks
  10. Reject dependency loops before anything can run
->

Artifact path

  1. If validation found errors, stop and return diagnostics
  2. Use the pipeline id to choose the artifact.duckdb location
  3. Look at the latest recorded run in the artifact database
  4. Do not compile over a run that is still active
  5. Clean up stale or failed run state when it is safe
  6. Compile again now that the exact artifact path is known
->

Hash contract

  1. Build a compile contract that describes this exact pipeline version
  2. Check project Python imports do not point outside the project
  3. Hash every project Python file from its file bytes
  4. Combine those file hashes into one project Python hash
  5. Hash whether config exists and the exact config text
  6. Normalize each node and hash its SQL, settings, outputs, and dependencies
  7. Sort dependency edges and hash the graph shape
  8. Collect runtime variable rules and reject conflicting usage
  9. Create one final contract hash from pipeline, nodes, edges, config, project files, and vars
->

Persist

  1. Create the compiled contract record used by run and inspect
  2. Store hashes, graph edges, tracked files, variables, spec, and diagnostics
  3. Open artifact.duckdb and save the contract metadata
  4. Write the row into the Queron metadata compiled contracts table
  5. Mark older compile contracts as inactive
  6. Make this compile the active contract for future run and inspect calls
Run Flowchart

Run request

  1. User calls run_pipeline with pipeline path and optional runtime inputs
  2. Queron emits early run and compile validation log events
  3. The pipeline file is compiled or validated before execution begins
  4. If compile validation fails, no nodes run and the result is compile_failed
->

Run safety

  1. Resolve the artifact.duckdb path for this pipeline
  2. Check the latest run stored in the artifact database
  3. Refuse to start if another run is still active
  4. If allowed, finalize or archive stale failed and orphaned running runs
  5. Check that a requested run label is not already used
->

Build runtime

  1. Create a PipelineRuntime for this run id
  2. Attach the compiled pipeline spec and module globals
  3. Open the artifact database path and working directory
  4. Validate runtime variables against the compiled contract
  5. Attach runtime database bindings and connections path
  6. Persist early log events into the runtime log stream
->

Prepare outputs

  1. If clean_existing is enabled, preserve incomplete outputs first
  2. Drop selected existing local artifact tables before running
  3. If target_node is set, select that node and all required upstream nodes
  4. Register this runtime as active so stop and force-stop can find it
->

Choose order

  1. Build a topological order from the compiled dependency graph
  2. Start with nodes that have no unmet dependencies
  3. Run check nodes before other ready nodes when possible
  4. Skip nodes outside the selected run scope
  5. A node only runs after all selected dependencies are complete
->

Start run

  1. Write a pipeline run row with status running
  2. Start the stop request watcher
  3. Create one node run row for each selected node
  4. Mark selected nodes as ready in node state history
->

Execute nodes

  1. Before each node, check whether a stop was requested
  2. Mark the node as running and record its start time
  3. Dispatch to the runtime handler for that node kind
  4. Ingress nodes load data into local DuckDB artifact tables
  5. Model SQL nodes read refs and write a new local artifact table
  6. Check nodes run validation queries and may fail the run
  7. Egress nodes read artifacts and write files or database targets
  8. Lookup nodes prepare database lookup tables for later SQL
->

Record outcome

  1. Normalize the node execution result
  2. Record row counts, artifact name, warnings, details, and finish time
  3. If warnings exist, mark the node complete_with_warnings
  4. If a node fails, mark it failed and skip downstream work when policy says so
  5. After each node, check again for stop requests at a safe boundary
->

Finish run

  1. When all selected nodes finish, mark the run success or success_with_warnings
  2. Archive completed local artifact tables for the run
  3. Clean up temporary lookup tables when retention does not keep them
  4. If execution failed, mark the run failed and preserve error details
  5. Unregister the active runtime and return executed nodes, run id, status, and log path
Run Functions

Each function below explains what happens to artifacts and node state.

compile_pipeline

How to call

  1. Python: queron.compile_pipeline(...)
  2. CLI: queron compile ...
->

Artifact use

  1. Reads pipeline.py and configurations.yaml
  2. Creates or updates artifact.duckdb for the pipeline
  3. Writes one active row into _queron_meta.compiled_contracts
  4. Marks older compiled contracts inactive
->

What the user gets

  1. A compiled pipeline object with diagnostics
  2. The artifact database path
  3. The compile id and hashes used by later runs
  4. No pipeline nodes execute during compile
run_pipeline

How to call

  1. Python: queron.run_pipeline(...)
  2. CLI: queron run ...
->

What happens

  1. Validates or recompiles the pipeline before execution
  2. Creates a new run id and a running row in artifact.duckdb
  3. Creates node run rows for the selected nodes
  4. Executes nodes in dependency order
  5. Returns executed nodes, artifact path, run id, status, and log path
->

Artifacts and nodes

  1. Each artifact-producing node writes to its compiled local table
  2. A target_node run executes that node plus required upstream nodes
  3. clean_existing drops selected existing artifact tables before execution
  4. Successful node rows store artifact name, row counts, warnings, and details
  5. Finished runs archive completed local artifact tables by run id
resume_pipeline

How to call

  1. Python: queron.resume_pipeline(...)
  2. CLI: queron resume ...
->

What happens

  1. Finds the latest failed or incomplete run in artifact.duckdb
  2. Validates the current compile contract before resuming
  3. Reuses stored runtime variables unless new values are provided
  4. Builds a runtime for the existing run context
  5. Runs only work that is still safe and necessary
->

Artifacts and nodes

  1. Completed artifact tables are kept instead of rebuilt
  2. Failed or skipped node states guide what needs to run again
  3. Previously completed nodes can stay complete when their artifacts exist
  4. New node outcomes are written back to the same artifact database
  5. The resumed run finishes with updated run and node status
stop_pipeline

How to call

  1. Python: queron.stop_pipeline(...)
  2. CLI: queron stop ...
->

What happens

  1. Finds the selected running run in artifact.duckdb
  2. Writes a graceful stop request file for that run
  3. The active runtime checks for the request at safe boundaries
  4. The current node is allowed to reach a safe stopping point
  5. Returns the request path, artifact path, run id, and stop mode
->

Artifacts and nodes

  1. Completed node artifacts remain in the artifact database
  2. Nodes not yet started can be marked skipped or left ready depending on policy
  3. The run status is updated when the runtime handles the stop
  4. Inspect can still read completed node outputs and logs
force_stop_pipeline

How to call

  1. Python: queron.force_stop_pipeline(...)
  2. CLI: queron force-stop ...
->

What happens

  1. Writes a force stop request for the selected running run
  2. The runtime tries registered interrupt callbacks for supported work
  3. Some connector or Python work may only stop at fallback boundaries
  4. The active node is converted into a stopped or failed outcome
  5. Returns the same stop result shape as graceful stop
->

Artifacts and nodes

  1. Artifacts from fully completed nodes remain usable
  2. The interrupted node may not produce a complete artifact
  3. Downstream nodes are not allowed to continue from incomplete inputs
  4. Node logs record whether interruption was attempted and where it happened
reset_node

How to call

  1. Python: queron.reset_node(...)
  2. CLI: queron reset-node ...
->

What happens

  1. Validates the current compiled pipeline
  2. Selects exactly the requested node
  3. Drops that node's local artifact table when it has one
  4. Clears that node's run state and marks it ready again
  5. Returns the reset node and dropped table names
->

Artifacts and nodes

  1. Only the selected node artifact is removed
  2. Upstream artifacts are left untouched
  3. Downstream artifacts may still exist, but can now be stale
  4. Use this when one node needs to be rerun without clearing the graph
reset_downstream

How to call

  1. Python: queron.reset_downstream(...)
  2. CLI: queron reset-downstream ...
->

What happens

  1. Validates the current compiled pipeline
  2. Finds the requested node and all downstream children
  3. Drops local artifact tables for the selected downstream set
  4. Resets those node states back to ready
  5. Returns all reset nodes and tables
->

Artifacts and nodes

  1. The selected node output is removed
  2. Every artifact that consumed it directly or indirectly is removed
  3. Upstream dependency artifacts are preserved
  4. Use this after changing or invalidating one producer node
reset_upstream

How to call

  1. Python: queron.reset_upstream(...)
  2. CLI: queron reset-upstream ...
->

What happens

  1. Validates the current compiled pipeline
  2. Finds the requested node and all upstream dependencies
  3. Drops local artifact tables for that upstream set
  4. Marks those node states ready for rebuild
  5. Returns all reset nodes and tables
->

Artifacts and nodes

  1. Input artifacts feeding the node are removed
  2. The selected node artifact is also removed when it exists
  3. Unrelated downstream branches are not selected by this reset
  4. Use this when source-side inputs need to be rebuilt
reset_all

How to call

  1. Python: queron.reset_all(...)
  2. CLI: queron reset-all ...
->

What happens

  1. Validates the current compiled pipeline
  2. Selects every node in the compiled graph
  3. Drops every local artifact table owned by the pipeline
  4. Marks all selected node states ready again
  5. Returns the full reset node and table list
->

Artifacts and nodes

  1. All local artifacts are removed from artifact.duckdb
  2. The compile contract and metadata database remain in place
  3. The next run rebuilds the pipeline from the beginning
  4. Use this when the local artifact state should be fully refreshed

_queron_meta.compiled_contracts

What it is

Stores the full compiled view of pipeline.py. This is not the raw Python file. It is Queron's saved understanding of the pipeline after decorators were read, SQL refs were resolved, dependencies were built, runtime vars were checked, and hashes were calculated.

What it is used for

Run, resume, reset, inspect, and graph UI start here. This row tells Queron which compile is active, where the artifact database lives, which nodes exist, how nodes depend on each other, what files affected the hash, what runtime variables are allowed, and whether compile produced diagnostics.

_queron_meta.pipeline_runs

What it is

Tracks each time a pipeline starts, finishes, fails, stops, or resumes.

What it is used for

Inspect commands and graph UI use it to show run status, labels, runtime vars, timing, and archived artifact location.

Sample record

{
  "run_id": "run_001",
  "run_label": "daily_load",
  "compile_id": "cmp_20260429_001",
  "pipeline_id": "customer_360_demo",
  "artifact_path": ".queron/customer_360_demo/artifact.duckdb",
  "archived_artifact_path": ".queron/customer_360_demo/run_001/artifact.duckdb",
  "started_at": "2026-04-29T14:01:00Z",
  "finished_at": "2026-04-29T14:03:15Z",
  "status": "success",
  "runtime_vars_json": {
    "as_of_date": "2026-04-29"
  },
  "is_final": true
}

_queron_meta.node_states

What it is

Keeps the state changes for each node, such as ready, running, complete, failed, skipped, or cleared.

What it is used for

Graph UI and inspect use it to show the current node state and how that state changed over time.

Sample record

{
  "node_state_id": "state_002",
  "run_id": "run_001",
  "node_run_id": "run_001:policy_core",
  "node_name": "policy_core",
  "state": "complete",
  "is_active": true,
  "created_at": "2026-04-29T14:02:20Z",
  "trigger": "node_completed",
  "details_json": {
    "artifact_name": "main.policy_core",
    "node_kind": "model.sql"
  }
}
_queron_meta.column_mapping

What it is

Stores how source columns were converted into DuckDB artifact columns.

What it is used for

Inspect node column views use it to explain source type, DuckDB type, mapping warnings, and lossy conversions.

Sample record

{
  "target_schema": "main",
  "target_table": "policy_core",
  "node_name": "policy_ingress",
  "node_kind": "postgres.ingress",
  "ordinal_position": 1,
  "source_column": "premium_amount",
  "source_type": "NUMERIC(12,2)",
  "target_column": "premium_amount",
  "target_type": "DECIMAL(12,2)",
  "connector_type": "postgres",
  "mapping_mode": "ingress",
  "warnings_json": [],
  "lossy": false
}
_queron_meta.table_lineage

What it is

Links a child artifact table to its source, lookup, or upstream artifact parents.

What it is used for

Graph and inspect use it to explain dependency meaning, not just dependency order.

Sample record

{
  "child_schema": "main",
  "child_table": "customer_summary",
  "parent_kind": "artifact",
  "parent_name": "policy_core",
  "parent_schema": "main",
  "parent_table": "policy_core",
  "connector_type": "duckdb",
  "via_node": "customer_summary"
}
main.<out>

What it is

These are the actual data tables produced by artifact-producing nodes. The name comes from the node's out value after Queron sanitizes it.

What it is used for

Later nodes read these tables through queron.ref(...), exports read from them, and inspect can preview them.

Sample record

{
  "table_name": "main.policy_core",
  "produced_by_node": "policy_core",
  "logical_out": "policy_core",
  "sample_row": {
    "policy_id": "P-1001",
    "customer_id": "C-501",
    "premium_amount": 128.5,
    "status": "active"
  }
}
run_<run_id>.<out>

What it is

When a run finishes, Queron can copy completed local artifacts into a run-specific archive database and schema.

What it is used for

Inspect can read the exact artifact data for a past run instead of only the current active tables.

Sample record

{
  "archived_artifact_path": ".queron/customer_360_demo/run_001/artifact.duckdb",
  "archived_artifact_name": "run_run001.policy_core",
  "original_artifact_name": "main.policy_core",
  "run_id": "run_001",
  "sample_row": {
    "policy_id": "P-1001",
    "customer_id": "C-501",
    "premium_amount": 128.5,
    "status": "active"
  }
}
User-Facing Artifact Functions

These functions read, write, inspect, or export data from artifact.duckdb and related run files.

inspect_dag

How to call

  • Python: queron.inspect_dag(...)
  • CLI: queron inspect_dag ...

What it is

Reads the compiled contract and selected run metadata to return the whole pipeline graph with node status.

What it is used for

Use it when you want to understand the full DAG, current run state, runtime var contract, and graph UI data.

    {
      "name": "as_of_date",
      "required": true
    }

    {
      "name": "policy_core",
      "kind": "model.sql",
      "status": "complete",
      "artifact_name": "main.policy_core",
      "dependencies": [
        "policy_ingress"
      ]

  "edges": [
    [
      "policy_ingress",
      "policy_core"
    ]

}
